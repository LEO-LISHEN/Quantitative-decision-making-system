本机当前使用 `8011` 时，在 `.env` 中配置：

```dotenv
APP_HOST=127.0.0.1
```

刷新成功后，推荐、Top20、自选股详情和 AI 股票上下文才会使用本次 Tushare 数据。未刷新或刷新失败时，系统回退到内置 Demo 快照。

### 前端说明

前端不是 React、Vue 或 Next.js，而是原生单页应用：

- 通过 URL Hash 切换页面，如 `#dashboard`、`#analysis`。
- 通过浏览器 `fetch()` 调用 FastAPI。
- 没有前端状态管理库，页面状态保存在 JavaScript 变量和 DOM 中。
- 没有图表库，行情趋势图由原生 SVG 绘制。
- 在线字体使用 Noto Sans SC 和 Inter，CDN 不可用时回退到系统字体。

## 9. 已知限制和技术债

### 高优先级

1. 当前所有改动尚未提交。接手人应先保护工作区，再整理提交。
2. Top20 不是全 A 股 Top20，只是固定 30 股样本池中的排序结果。
3. Tushare 缓存仅在内存，服务重启即丢失，需要手动刷新。
4. 自选股只存本机 JSON，没有用户体系、数据库和多用户隔离。
5. 分析任务也保存在进程内存，重启后任务状态丢失。
6. Demo 快照只有少量股票；无 Tushare 时，搜索和 Top20 能力会明显缩小。

### 中优先级

1. 多因子模型是演示级规则，没有回测、交易成本、停牌/ST/涨跌停过滤和因子有效性验证。
2. 质量因子受 Tushare 接口权限影响，不可用时使用中性分。
3. AI 问答的“知识库”只是代码内静态列表，不是 RAG。
4. 盘前、盘后报告目前是 Demo 摘要，不是定时生成的真实报告。
5. 企业微信推送仅提供测试入口，未实现提醒条件、任务调度和消息队列。
6. Event Focus 依赖公共 RSS 和外部模型，来源稳定性、合规性和时效性仍需评估。
7. Docker Compose 仍包含 PostgreSQL 方案，但当前产品路线已改为 Demo 直接使用 Tushare，编排文件没有作为当前主启动方式再次完整验收。

### 代码维护

1. 前端单个 `app.js` 已超过 1200 行，后续应按页面拆分。
2. `styles.css` 与 `terminal-theme.css` 存在覆盖关系，长期应合并设计变量并清理旧样式。
3. `app/demo_service.py` 同时承担数据、推荐、自选、问答、通知，后续应按领域拆分。
4. 缺少 API 集成测试、浏览器自动化测试和持续集成。
5. 部分外部请求使用标准库直接调用，超时、重试、限流和日志仍较简单。
6. `Financial Analyst Skills` 历史上可能包含独立 `.git` 元数据，合并仓库前需确认 Git 跟踪方式。

## 10. 建议的后续开发顺序

1. **先保护成果**：检查未跟踪文件、删除本机敏感信息、创建分支并提交当前工作区。
2. **稳定启动流程**：统一端口和 README，增加启动时自动刷新或后台刷新，补健康检查。
3. **扩大股票池**：从固定 30 股改为可配置股票池，增加 Tushare 限流、分批请求和本地缓存。
4. **持久化应用状态**：优先保存自选股、提醒规则、分析任务和数据刷新记录；再决定使用 PostgreSQL 还是轻量 SQLite。
5. **补齐真实报告**：将盘前、盘后报告连接到推荐、持仓、自选和事件数据，并支持定时生成。
6. **实现盯盘提醒**：定义价格、涨跌幅、均线、事件和风险阈值，接入调度与企业微信推送。
7. **升级知识库**：将产品规则、研究方法和报告做分块检索，再接向量库；不要直接把整个文档塞给模型。
8. **验证策略**：增加历史回测、基准比较、换手率、交易成本、最大回撤和避免未来函数检查。
9. **拆分前端和服务层**：业务稳定后再考虑 React/Next.js，不建议为了技术栈形式立即重写。
10. **补 CI/CD**：至少自动运行 Python 测试、JS 语法检查和 API 冒烟测试。
